# HireSight Deployment Guide

This project is deployed as two services:

- **Backend and AI pipeline:** Render (Docker)
- **Frontend:** Vercel (Vite)
- **Database and file storage:** Firebase
- **AI APIs:** Groq

The `AI/` directory does not need a separate deployment. Render includes it
inside the backend Docker image.

## 1. Prepare the repository

Before deploying, make sure your latest code is committed and pushed to GitHub.

```powershell
git status
git add .
git commit -m "Prepare HireSight for deployment"
git push origin main
```

Do not commit any `.env` file or Firebase service-account JSON. The repository's
`.gitignore` and `.dockerignore` already exclude these files.

If a real API key or Firebase credential was previously committed to a public
repository, revoke it and generate a replacement before continuing.

## 2. Create the Firebase resources

1. Open [Firebase Console](https://console.firebase.google.com/).
2. Create a project or open the project used by HireSight.
3. Open **Build > Firestore Database** and create a database.
4. Open **Build > Storage** and create the default storage bucket.
5. Copy the exact bucket name shown in Firebase Storage. It may look like:
   `your-project.appspot.com` or `your-project.firebasestorage.app`.
6. Open **Project settings > Service accounts**.
7. Select **Generate new private key** and download the JSON file.
8. Keep this JSON private. It will be stored as a Render secret, not uploaded
   to GitHub.

Convert the downloaded service-account JSON into a single line before putting
it into Render:

```powershell
Get-Content "C:\path\to\service-account.json" -Raw |
  ConvertFrom-Json |
  ConvertTo-Json -Compress
```

Copy the command output. Do not save or commit it inside this repository.

## 3. Create a Groq API key

1. Open the [Groq Console](https://console.groq.com/keys).
2. Create an API key.
3. Copy it somewhere secure; it will be added only to Render.

## 4. Deploy the backend to Render

The repository already contains `render.yaml` and `backend/Dockerfile`, so use
a Render Blueprint.

1. Sign in to [Render](https://dashboard.render.com/).
2. Select **New > Blueprint**.
3. Connect the GitHub repository:
   `https://github.com/iam-ankur01/Hackathon`
4. Select the `main` branch.
5. Render should detect the root-level `render.yaml`.
6. Confirm creation of the `hiresight-api` web service.
7. Enter the secret environment variables requested by the Blueprint:

| Variable | Value |
|---|---|
| `GROQ_API_KEY` | Your Groq API key |
| `FIREBASE_CREDENTIALS_JSON` | The compressed, one-line service-account JSON |
| `FIREBASE_STORAGE_BUCKET` | The exact Firebase Storage bucket name |
| `FRONTEND_ORIGIN` | Temporarily use `https://example.com` |

The Blueprint automatically sets these additional values:

| Variable | Value |
|---|---|
| `APP_ENV` | `production` |
| `USE_LOCAL_DB` | `false` |
| `SECRET_KEY` | Secure value generated by Render |

8. Start the deployment.
9. Wait for the service status to become **Live**.
10. Copy its public URL, for example:
    `https://hiresight-api.onrender.com`

The configured Render plan is `starter`, which is a paid always-on instance.
To use another available plan, edit the service plan in Render or change
`plan` in `render.yaml` before deployment.

### Verify the backend

Open these URLs in a browser, replacing the example hostname:

- `https://hiresight-api.onrender.com/health`
- `https://hiresight-api.onrender.com/ready`
- `https://hiresight-api.onrender.com/docs`

Expected health response:

```json
{"status":"healthy"}
```

Expected readiness response:

```json
{"status":"ready"}
```

If `/health` works but `/ready` fails, check the Firebase credentials,
Firestore setup, and storage bucket environment variable in Render.

## 5. Deploy the frontend to Vercel

1. Sign in to [Vercel](https://vercel.com/).
2. Select **Add New > Project**.
3. Import the same GitHub repository.
4. Configure the project:

| Setting | Value |
|---|---|
| Framework preset | Vite |
| Root directory | `Frontend` |
| Build command | `npm run build` |
| Output directory | `dist` |

5. Add this environment variable for **Production**, **Preview**, and
   **Development**:

| Variable | Value |
|---|---|
| `VITE_API_URL` | Your Render backend URL, with no trailing slash |

Example:

```text
VITE_API_URL=https://hiresight-api.onrender.com
```

6. Select **Deploy**.
7. When deployment finishes, copy the production URL, for example:
   `https://hiresight.vercel.app`.

The existing `Frontend/vercel.json` sends React Router paths back to
`index.html`, so refreshing pages such as `/dashboard` should work.

## 6. Allow the Vercel frontend through backend CORS

1. Return to the `hiresight-api` service in Render.
2. Open **Environment**.
3. Replace the temporary `FRONTEND_ORIGIN` value with the exact Vercel URL:

```text
https://hiresight.vercel.app
```

4. Do not add a trailing slash.
5. Save the environment change and allow Render to redeploy the service.

To support multiple frontend domains, use a comma-separated value:

```text
https://hiresight.vercel.app,https://www.yourdomain.com
```

Vercel preview deployments use changing URLs. Production is simplest with only
the stable production URL allowed.

## 7. Run an end-to-end production test

From the Vercel URL:

1. Create a new account.
2. Log out and log back in.
3. Update the profile.
4. Upload a small PDF or DOCX resume.
5. Upload a short interview recording.
6. Wait for the AI evaluation to complete.
7. Open the generated report, dashboard, progress, roadmap, and coach pages.
8. Refresh a nested page to verify Vercel routing.

Also check:

- Render **Logs** for backend, Firebase, or Groq errors.
- Browser developer tools **Network** tab for CORS or API errors.
- Firebase Firestore for newly created user and interview documents.
- Firebase Storage for uploaded files.

## 8. Automatic future deployments

After initial setup:

- A push to `main` automatically redeploys the Render backend.
- A push to `main` automatically redeploys the Vercel production frontend.
- Pull requests can receive Vercel preview deployments.

If `VITE_API_URL` changes, update it in Vercel and redeploy the frontend.
If the frontend domain changes, update `FRONTEND_ORIGIN` in Render and redeploy
the backend.

## Troubleshooting

### Render reports an invalid production configuration

Confirm that all required variables are populated:

- `SECRET_KEY`
- `GROQ_API_KEY`
- `FIREBASE_CREDENTIALS_JSON`
- `FIREBASE_STORAGE_BUCKET`
- `FRONTEND_ORIGIN`
- `USE_LOCAL_DB=false`

`FRONTEND_ORIGIN` must begin with `https://` in production.

### Firebase JSON parsing fails

The `FIREBASE_CREDENTIALS_JSON` value must contain the complete service-account
JSON. Compress it to one line using the PowerShell command in step 2 and paste
the output without surrounding quotes.

### The browser shows a CORS error

Make sure `FRONTEND_ORIGIN` exactly matches the URL in the browser, including
the `https://` prefix and excluding a trailing slash. Redeploy Render after
changing the variable.

### The frontend still calls localhost

Check that `VITE_API_URL` is set in Vercel, then trigger a new frontend
deployment. Vite inserts environment variables during the build, so changing
the variable does not alter an already-built deployment.

### Audio or video analysis fails

Check Render logs for the Groq error. Confirm that:

- `GROQ_API_KEY` is valid.
- The uploaded format is supported.
- The Render instance has enough memory and request time for the recording.
- The Groq account has available quota.

The backend Docker image already installs `ffmpeg`.
